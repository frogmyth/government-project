# AI 흔적 방지 규칙 (한국어 정부지원사업 문서 특화)

> 이 규칙은 AI가 작성한 정부지원사업 계획서에서 AI 생성 흔적을 제거하고,
> 실무 경험이 있는 사람이 직접 작성한 것처럼 보이도록 교정하는 기준입니다.
>
> **v1.1 업데이트**: 실제 작성완료 문서(서식1 기술개요서, 서식4 연구개발계획서 등)와
> AI 초안(drafts/) 비교 분석을 통해 규칙을 보강하였습니다.

---

## 1. 이모지 및 특수문자 규칙

### 절대 금지
- 제목, 소제목, 본문에 이모지 사용 금지 (🚀🤖🎨🌱📊 등)
- 정부 공식 서류에 이모지를 쓰는 실무자는 없음
- ✅/❌ 체크마크 → "O/X" 또는 "지원/미지원"으로 대체

### 허용
- 서식 기호: ◦, -, ※, ·
- 화살표: →, ↔ (기술 흐름도에서만)
- 괄호: (), [], 「」
- 체크박스 표기: [ V ], [  ] (정부 서식 선택란에만 사용)

### 실제 사례 (drafts → 작성완료 비교)
- AI 초안: `## 2. 핵심 기술 목표 — AI 기능 강화 🚀` → 작성완료: `## 2. 핵심 기술 목표`
- AI 초안: `✅ 운영 중` → 작성완료: 별도 표시 없이 "운영 중" 텍스트만 사용
- AI 초안: 섹션 제목마다 🤖🔄🎨🗣️📅🗺️📋 등 이모지 → 작성완료: 전부 삭제

## 2. 수식어 및 과장 표현 규칙

### AI 전형적 패턴 (수정 대상)
| AI가 쓰는 표현 | 사람이 쓰는 표현 |
|---------------|----------------|
| 세계 최초 (3회 이상 반복) | 최초 1회만 사용, 이후 "차별화된", "독자적인" |
| 세계 유일 | "현재까지 유사 사례가 보고되지 않은" |
| 유일무이 | "경쟁 제품 대비 독보적인" |
| 혁신적인 | 삭제하거나 구체적 내용으로 대체 |
| 획기적인 | "기존 대비 개선된" |
| 최첨단 | "최신" 또는 구체적 기술명 |
| 원천 기술 | "핵심 기술" (원천 기술은 근거 필요) |
| 비약적인 성장 | 구체적 수치로 대체 ("전년 대비 40% 성장") |
| 폭발적인 수요 | "수요가 빠르게 증가하고 있는" |

### 수식어 사용 한도
- "세계 최초/최초" 계열: 문서 전체에서 최대 2회
- "유일/독자" 계열: 문서 전체에서 최대 2회
- 같은 수식어가 2페이지 이내에 재등장하면 삭제

### 실제 사례 (drafts → 작성완료 비교)
- AI 초안: "세계 최초"가 10회 이상 등장 + "세계 유일", "유일무이", "업계 최초" 난발
  - `**세계 유일** 생성형 AI 기반 리테일 콘텐츠 자동 생성`
  - `**세계 최초** 대화형 챗봇 CMS 컨트롤`
  - `**업계 최초** AI 기반 롤링 최적화`
  - `**유일무이** Battery-Free 레일 전원`
- 작성완료: "세계 최초" 1회만 사용, 나머지는 구체적 기술 설명으로 대체
- AI 초안: `"세계 최초, 세계 최고의 AI 리테일 솔루션으로 대한민국이 글로벌 리테일테크를 선도하겠습니다."` → 마케팅 슬로건 톤 (정부 제안서에 부적절)

## 3. 문장 구조 규칙

### 문장 길이 불균일화
AI는 비슷한 길이의 문장을 나열하는 경향이 있음. 이를 방지하기 위해:
- 짧은 문장(10자 이내)과 긴 문장(40자 이상)을 섞어 사용
- 3개 연속으로 같은 길이대의 문장이 오지 않도록 조정

### bullet 구조 변형
AI 패턴:
```
- A 기술을 활용하여 B를 구현합니다
- C 기술을 활용하여 D를 구현합니다
- E 기술을 활용하여 F를 구현합니다
```

인간화 수정:
```
- A 기술 기반으로 B를 구현한다
- C 기술과 D를 결합하여 적용하며, 이를 통해 기존 시스템 대비 성능을 높인다
- E에서 F로의 전환은 단계적으로 진행할 예정이다
```

### 문체 혼용
- 한 섹션 안에서 "~합니다"체와 "~한다"체를 적절히 섞을 수 있음
- 다만 같은 문단 내에서는 통일
- 표 안의 문장은 간결체("~함", "~예정") 사용 가능

### 실제 사례 (drafts → 작성완료 비교)
- AI 초안의 bullet 반복 패턴:
  ```
  - AI가 상품별 조회수 분석 → 최적 순서 자동 결정
  - 고객 체류 시간 분석 → 동적 간격 조정
  - 위치별 맞춤 콘텐츠 (계산대/입구/냉장고 앞)
  ```
  → 모든 bullet이 "명사구 → 결과"의 동일 패턴
- 작성완료에서는 bullet 일부를 서술형으로 풀어 쓰거나, ◦ 기호와 - 기호를 섞어 사용

## 4. 접속사 다양화 규칙

### AI 반복 패턴 → 교체 표현
| 반복되는 접속사 | 교체 후보 |
|---------------|----------|
| 또한 | 아울러, 이에 더해, 한편, 더불어, 나아가 |
| 따라서 | 이에 따라, 결과적으로, 이러한 배경에서 |
| 특히 | 그중에서도, 무엇보다, 주목할 점은 |
| 이를 통해 | 이를 바탕으로, 이 과정에서, 이러한 방식으로 |
| 구체적으로 | 좀 더 살펴보면, 세부적으로는, 실제로는 |

### 접속사 사용 한도
- 같은 접속사가 한 페이지(약 500자) 안에 2회 이상 나오면 교체

## 5. 나열 및 병렬 구조 규칙

### 3의 법칙 회피
AI는 "첫째, 둘째, 셋째" 또는 3개씩 나열하는 경향이 강함.
- 때로는 2개만, 때로는 4~5개를 나열
- "첫째/둘째/셋째" 대신 바로 내용 서술

### 표 내용 불균일
AI는 표의 모든 셀을 빈틈없이 채움.
- 일부 셀에 "-" 또는 "확인 중" 또는 "해당없음"을 넣으면 자연스러움
- 비고란에 모든 행마다 내용을 넣지 않아도 됨

## 6. 볼드/강조 규칙

### AI 패턴
모든 핵심어에 **볼드** 처리 → 정작 뭐가 중요한지 구분 불가

### 인간화 규칙
- **볼드**는 한 섹션에서 최대 3~4개만 사용
- 정말 심사위원이 봐야 할 핵심 수치나 차별점에만 적용
- 과제명, 기관명 등 고유명사는 첫 등장 시에만 볼드

## 7. 코드 블록 규칙 (정부 제안서용)

### 원칙
정부지원사업 계획서에 Python/JavaScript 코드를 넣는 실무자는 거의 없음.

### 처리 방법
- 코드 블록 → 자연어 설명 또는 순서도(텍스트)로 변환
- 알고리즘 설명 → "입력 데이터를 분석하여 ~을 산출하고, 이를 기반으로 ~를 결정하는 방식으로 동작한다"
- 기술 구현 상세는 표 형태로 정리

### 예외
- 시스템 구성도(ASCII 박스 다이어그램)는 유지 가능
- API 엔드포인트 목록 등 기술 명세는 표로 정리

### 실제 사례 (drafts → 작성완료 비교)
- AI 초안에 Python 코드가 9개 이상 포함:
  - `class AIRollingOptimizer:` (롤링 최적화)
  - `def suggest_motion(product_category):` (동영상 움직임 제안)
  - `def recommend_sale_timing(product_id):` (세일 추천)
  - `class AdaptiveProcessingDistributor:` (분산 처리 특허)
  - `class TouchIntentClassifier(nn.Module):` (터치 분류)
  - `class VoiceIntentExtractor:` (음성 의도 추출)
  등 AI가 코드로 자신의 능력을 과시하는 전형적 패턴
- 작성완료: 코드 전부 삭제 → 절차 설명을 텍스트 흐름도(`[단계1] → [단계2] → ...`)로 변환
- 유지된 코드: Cloud-Edge-Device 시스템 구성도(ASCII 박스)만 유지

## 8. 반복 내용 통합 규칙

### 점검 대상
AI는 같은 내용을 여러 섹션에서 반복하는 경향이 있음:
- "핵심 차별점" 목록이 3곳 이상에서 반복
- 매출 목표 표가 2곳 이상에서 동일하게 등장
- 경쟁사 비교표가 유사 형태로 반복

### 처리 방법
- 처음 등장하는 곳에서 상세 기술
- 이후 등장 시 "앞서 기술한 바와 같이" 또는 "상기 표 참조"로 축약
- 완전히 동일한 표가 2회 이상이면 1회만 유지

## 9. 정부 문서 톤 규칙

### 적절한 톤
- 격식체 사용 ("~합니다", "~입니다")
- 다만 지나치게 딱딱하면 안 됨 → 적절한 서술형 문장 포함
- 객관적 서술: "~할 것으로 기대된다", "~할 수 있을 것으로 판단된다"

### 피해야 할 톤
- 마케팅 카피 톤: "당신의 매장을 혁신하세요!"
- 블로그 톤: "오늘은 ~에 대해 알아보겠습니다"
- 교과서 톤: "~이란 ~을 의미한다"

### 실무자 느낌 살리기
- "현재 자사에서 운영 중인 CMS에서는..." (운영 경험 드러내기)
- "올리브영 파일럿 도입 결과..." (실제 사례 언급)
- "기존 시스템 운영 과정에서 확인된 한계점은..." (실무 기반 서술)

## 10. 숫자 및 데이터 표현 규칙

### AI 패턴
- 모든 수치에 근거를 붙이려 함 → 때로는 출처 없이 수치만 제시해도 됨
- 지나치게 정밀한 수치: "22.1%" → 실무에서는 "약 22%"도 자연스러움

### 인간화 규칙
- 핵심 수치만 정밀하게, 나머지는 "약", "대략" 허용
- 모든 숫자에 출처를 달 필요 없음 (핵심 시장 데이터만 출처 명시)
- "CAGR 22.1% (출처: MarketsandMarkets)" → 적절
- "직원 4명 (출처: 사업자등록증)" → 과도함, 출처 삭제

```
## 1. 개발기술 개요
### 1-1. 개발기술의 필요성
### 1-1-1. 개발 배경
```

## 12. 정부 제안서 특화 규칙

### 심사위원 관점 고려
- 심사위원은 수십 건을 검토함 → 핵심이 빨리 보여야 함
- AI가 쓴 듯한 뻔한 문장은 감점 요인이 될 수 있음
- 회사만의 실제 경험, 구체적 사례가 드러나야 신뢰감 상승

### 구체성 강화
- "AI 기술을 활용하여" → "Llama 3 8B 모델에 자사 리테일 데이터 10만 건을 학습시켜"
- "성능이 향상됩니다" → "기존 120초 소요되던 콘텐츠 생성이 25초 이내로 단축됩니다"

### 자사 경험 반영
- "~할 수 있습니다" (가능성) → "자사 CMS 운영 경험상 ~한 것을 확인하였습니다" (경험)
- 올리브영, CU 등 실제 도입 사례를 적극 활용
- "기대됩니다" 일변도가 아니라 "실증을 통해 확인된 바 있습니다" 혼용

## 13. 허위/과장 경력 및 수치 규칙

### AI 패턴
AI는 기업 정보를 "더 좋아 보이게" 각색하는 경향이 있음:
- 학력/경력을 실제와 다르게 기재
- 수치를 과대 포장하거나 출처 없는 수치를 삽입

### 실제 사례 (drafts → 작성완료 비교)
- AI 초안: `김현학 - 연세대학교 전기전자공학과 박사, 삼성전자 종합기술원, 네이버 클로바`
- 작성완료: `김현학 - 고려대학교 공학박사, 삼성전기 ESL 그룹장, ㈜라인어스 대표이사`
  → AI가 "더 화려한" 경력으로 변조함 (연세대→고려대, 삼성전자→삼성전기)
- AI 초안: `SCI급 15편 (1저자 8편), 특허 3건`
- 작성완료: `논문 25편, 특허 국내 47건/해외 62건`
  → 수치가 완전히 다름

### 인간화 규칙
- **절대 원칙**: 경력, 학력, 수상, 특허 수 등 팩트 데이터는 반드시 원본 확인 후 기재
- COMPANY_INFO.md 또는 참여연구원_주요업적.md의 데이터를 정본(正本)으로 사용
- AI가 생성한 경력/수치는 반드시 교차 검증

## 14. 정부 서식 체크란 규칙

### AI 패턴
AI는 정부 서식의 체크박스/선택란 형식을 무시하고 자유 서술하는 경향이 있음.

### 인간화 규칙
- 정부 서식에 체크란이 있으면 `[ V ]` / `[  ]` 형태로 명확히 표기
- 단답형 항목(사업자등록번호, 법인등록번호 등)은 표 형태로 정리
- "해당없음"은 구체적 사유와 함께 기재 (예: `**해당없음** (단독연구개발)`)

### 실제 사례 (작성완료 문서 패턴)
```
### 개발유형 (선택 1)
- [ V ] 단독연구개발
- [  ] 공동연구개발
```
→ 정부 hwpx 서식을 그대로 반영한 형태. AI 초안에서는 이런 형식을 쓰지 않고 산문체로 서술함.

## 15. 미완성/불확실 정보 표기 규칙

### AI 패턴
AI는 모든 항목을 확정된 것처럼 채워 넣음. 빈 칸이나 "확인 필요"를 남기지 않음.

### 인간화 규칙
- 확인되지 않은 정보는 솔직하게 표기: `확인 필요`, `추후 확인`, `XXX-XXXX`
- "확인 필요 사항" 섹션을 별도로 두어 미완성 항목을 명시

### 실제 사례 (작성완료 문서)
```
# ⚠️ 확인 필요 사항
1. ✅ **사업자등록번호** - 472-81-03313 (확인완료)
2. [ ] **연구책임자 국가연구자번호** - IRIS 시스템에서 발급/조회
3. [ ] **연락처** - 정확한 전화번호/이메일 확인
```
→ 실무자는 모르는 것을 모른다고 솔직하게 표기함. AI는 이런 항목도 그럴듯하게 채워 넣음.

## 16. 규정 적합성 자가 검증 규칙

### AI 패턴
AI는 예산표를 만들면서 규정 준수 여부를 검증하지 않음.

### 인간화 규칙
- 연구개발비 비목별 비율 한도를 명시적으로 검증
- 검증 결과를 표로 정리하여 "규정 적합성 체크" 섹션으로 포함

### 실제 사례 (작성완료: 연구개발비_총괄표.md)
```
| 규정 | 산출 | 판정 |
|------|------|:----:|
| 인건비 ≤ 총연구개발비 50% | 100,000/200,000 = 50.0% | O |
| 연구수당 ≤ 총연구개발비 2.5% | 5,000/200,000 = 2.5% | O |
```
→ 실무자가 직접 계산하고 규정 준수를 확인한 흔적. AI는 이런 자가 검증을 하지 않음.

## 17. 연차별 차이 설명 규칙

### AI 패턴
AI는 1차년도와 2차년도 예산/계획을 동일한 구조로 복사-수정하면서 차이의 이유를 설명하지 않음.

### 인간화 규칙
- 연차별 예산 차이가 있으면 이유를 한 줄씩 기재
- "왜 달라졌는지"를 자연어로 설명

### 실제 사례 (작성완료: 연구개발비_총괄표.md)
```
## 1차↔2차 년도 차이점
- **연구시설·장비비**: 1차(30,000) → 2차(20,000) — 초기 장비 집중 투자
- **연구재료비**: 1차(25,000) → 2차(30,000) — 2차년도 실험·테스트 확대
```
→ 짧은 한 줄 메모지만, 심사위원에게 예산 설계의 의도가 전달됨.

## 18. 참조 연결 규칙

### AI 패턴
AI는 모든 정보를 해당 섹션에 전부 풀어 씀 → 문서가 길어지고 반복됨.

### 인간화 규칙
- 다른 서식/문서에 이미 있는 내용은 참조로 연결
- 예: `[서식1] 기술개요서 참조`, `상기 표 참조`, `경쟁사_플래노그램_상세분석.md 참조`

### 실제 사례 (작성완료: 서식4 연구개발계획서)
```
## [1]~[10] 기술개요서와 동일
> `[서식1] 기술개요서_작성내용.md` 참조
```
→ 중복을 피하고 문서 간 연결을 활용. AI는 같은 내용을 반복 기술하려 함.
